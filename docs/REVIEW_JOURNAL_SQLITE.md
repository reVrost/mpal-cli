# SQLite Review Journal

The SQLite review journal is a durable decision ledger. It is not a cache and
should not store every intermediate `mpal` command, raw source packet, private
policy file, or private chain-of-thought.

Every `mpal strategy run` now stores the deterministic model packet as the
first-pass review row and returns `journal_entry_id` in the strategy output:

```text
model output -> agent recommendation -> human final call
```

Use the CLI flow:

```sh
mpal strategy run --date YYYY-MM-DD --universe universe.json --portfolio portfolio.json --config strategies/example.yaml --json
mpal report review_... --json
mpal journal finalize --id review_... --input tmp/mpal-runs/trade-review-finalize.json --json
```

The database path is:

```text
~/.marketpal/mpal.db
```

## Tables

The v1 schema intentionally uses two tables:

- `trade_reviews`: one row per reviewed packet.
- `trade_review_positions`: one row per ticker considered in that review.

The schema lives in:

```text
pkg/mpal/sqlitejournal/schema.sql
```

SQL accessors are generated by `sqlc` from:

```text
pkg/mpal/sqlitejournal/queries.sql
```

## What Belongs Here

Keep the SQLite journal focused on durable decisions:

- reviewed strategy id and full strategy config text;
- reviewed universe ticker array;
- user-requested ticker array;
- execution result;
- deterministic sizing fields including raw Kelly, fractional Kelly, Kelly
  target weight, final target weight, estimated value, and binding constraint;
- agent harness/model/skill and prompt/chat context;
- final human decision and reasoning;
- final validation result;
- one row per ticker showing model bucket, agent decision, and human decision.

Do not store API keys, broker credentials, private policy file contents, or raw
API/cache payloads. Prompt and rationale fields should contain concise
decision context, not hidden model reasoning transcripts.

## What Stays Outside

These should remain temporary artifacts or future cache data unless summarized
into a real review:

- ticker events calls;
- ticker financials calls;
- Markov calls;
- watchlist reads;
- portfolio snapshots;
- doctor checks;
- broad DD source packs;
- ticker-level exploratory commands that are not tied to a strategy run.

HTML reports should remain deterministic local files. SQLite stores only the
report path for the reviewed packet after `mpal report <trade_review_id>`.

For the broader provenance model, including timestamp meanings, config hashes,
freshness fields, deterministic versus API-derived fields, and local storage
boundaries, see [DATA_PROVENANCE_AND_AUDIT.md](DATA_PROVENANCE_AND_AUDIT.md).
